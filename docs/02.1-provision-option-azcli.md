# 02.1 (옵션) az aks CLI로 인프라 프로비저닝

> 🟢 **실행** = 직접 입력·수행 · 👁️ **예시** = 눈으로만(개념/발췌) · 📋 **예상 출력** = 비교용(입력 불필요)

> 🔀 **이 문서는 [02. 인프라 프로비저닝(Terraform)](02-provision-terraform.md)의 대체 옵션입니다.**
> **기본 경로는 Terraform**이며, 보통은 02 문서를 그대로 따르면 됩니다. 이 문서는 **Terraform이 잘 동작하지 않을 때**의 우회 경로입니다. 예를 들어:
> - `terraform init`에서 프로바이더(azurerm) 다운로드 실패(네트워크/프록시/레지스트리 차단)
> - 상태 파일(`terraform.tfstate`) 손상·잠금·충돌
> - 사내 정책상 Terraform 사용이 제한되는 환경
> - 그 밖에 Terraform `apply`가 반복 실패해 진행이 막히는 경우
>
> 위와 같은 상황이 아니라면 **02 문서(Terraform)를 우선 사용하세요.** 이 문서는 동일한 결과를 **Azure CLI만으로** 만드는 방법을 제공합니다.

Azure CLI만으로 Terraform 구성(`main.tf`/`network.tf`/`aks.tf`/`monitoring.tf`)과 **같은 결과**(3개 RG·BYO VNet·ACR·AKS+NAP+KEDA·Managed Prometheus/Grafana·역할 할당)를 재현합니다.

- 예상 소요: 약 10~15분(AKS 생성이 대부분)
- 사전 조건: [01. 사전 준비](01-prerequisites.md)를 완료한 상태(로그인·구독 선택·공급자 등록·쿼터 확인). 저장소 클론도 동일하게 필요합니다.

> ⚠️ **주의:** 이 경로는 Terraform 상태(state)를 만들지 않습니다. 이후 모듈(03~09)에서 `terraform output`으로 값을 읽는 부분은 아래에서 설정하는 **셸 변수(`$RG`, `$AKS` 등)** 로 대체하세요. 정리도 `terraform destroy` 대신 **`az group delete`** 로 수행합니다([10. 정리](10-cleanup.md)의 1-A 참고).

## (0) 공통 변수 설정

Terraform의 명명 규칙(`CAF`)·난수 접미사를 그대로 따릅니다.
🟢 **실행**
```bash
# 기본값(필요 시 수정). SUFFIX는 전역 고유 이름 충돌을 막는 5자리 숫자.
PREFIX=akshol
LOCATION=koreacentral
REGION_SHORT=krc
ENVIRONMENT=dev
WORKLOAD=aksworkshop
SUFFIX=$(( RANDOM % 90000 + 10000 ))   # 예: 37291
VM_SIZE=Standard_D2s_v5   # 데모 최소사양(운영은 D4s_v5+)
NODE_COUNT=2

NAME="${PREFIX}${SUFFIX}"
RG_SUFFIX="${ENVIRONMENT}-${REGION_SHORT}-${SUFFIX}"

# 리소스 그룹(3종) / 리소스 이름
RG_NET="rg-${WORKLOAD}-network-${RG_SUFFIX}"
RG="rg-${WORKLOAD}-aks-${RG_SUFFIX}"          # 워크로드(AKS/ACR) RG
RG_MON="rg-${WORKLOAD}-monitoring-${RG_SUFFIX}"
VNET="vnet-${WORKLOAD}-${ENVIRONMENT}-${REGION_SHORT}-${SUFFIX}"
SUBNET="snet-aks-${ENVIRONMENT}"
ACR="acr${NAME}"
AKS="aks-${NAME}"
LAW="law-${NAME}"
AMW="amw-${NAME}"
GRAFANA="graf-${NAME}"

# 공통 태그(WAF 운영 우수성)
TAGS="workload=${WORKLOAD} environment=${ENVIRONMENT} owner=platform-team costCenter=workshop managedBy=azcli"

echo "AKS=$AKS  RG=$RG  (suffix=$SUFFIX)"
```

## (1) 리소스 그룹 3종 + BYO 네트워크

Terraform의 `main.tf`/`network.tf`에 대응.
🟢 **실행**
```bash
az group create -n "$RG_NET" -l "$LOCATION" --tags $TAGS layer=network
az group create -n "$RG"     -l "$LOCATION" --tags $TAGS layer=workload
az group create -n "$RG_MON" -l "$LOCATION" --tags $TAGS layer=monitoring

# VNet(10.224.0.0/16) + AKS 노드 서브넷(10.224.0.0/24)
az network vnet create -g "$RG_NET" -n "$VNET" -l "$LOCATION" \
  --address-prefixes 10.224.0.0/16 \
  --subnet-name "$SUBNET" --subnet-prefixes 10.224.0.0/24 --tags $TAGS

SUBNET_ID=$(az network vnet subnet show -g "$RG_NET" --vnet-name "$VNET" -n "$SUBNET" --query id -o tsv)
```

## (2) ACR + 모니터링 백엔드(Log Analytics / AMW / Grafana)

`aks.tf`(ACR), `monitoring.tf`에 대응.
🟢 **실행**
```bash
# ACR (Basic, admin 비활성)
az acr create -g "$RG" -n "$ACR" -l "$LOCATION" --sku Basic --admin-enabled false --tags $TAGS

# Log Analytics(Container Insights 로그 저장소)
LAW_ID=$(az monitor log-analytics workspace create -g "$RG_MON" -n "$LAW" -l "$LOCATION" \
  --retention-time 30 --tags $TAGS --query id -o tsv)

# Azure Monitor Workspace(Managed Prometheus 메트릭 저장소)
AMW_ID=$(az monitor account create -g "$RG_MON" -n "$AMW" -l "$LOCATION" --tags $TAGS --query id -o tsv)

# Managed Grafana (v12, SystemAssigned ID) — AMW 통합 연결은 아래 (4)에서 수행
az grafana create -g "$RG_MON" -n "$GRAFANA" -l "$LOCATION" \
  --grafana-major-version 12 --tags $TAGS
GRAFANA_ID=$(az grafana show -g "$RG_MON" -n "$GRAFANA" --query id -o tsv)
```
> `az grafana` 명령에는 `amg` 확장이 필요합니다. 없으면 `az extension add -n amg` 후 진행하세요.

## (3) AKS 클러스터 생성

(+ CNI Overlay/Cilium + KEDA + Container Insights + ACR 연결) — `aks.tf`의 핵심. Managed Prometheus 메트릭은 클러스터 생성 시점이 아니라 **아래 (5) 배포 마지막 단계**에서 `az aks update`로 활성화합니다.
🟢 **실행**
```bash
az aks create -g "$RG" -n "$AKS" -l "$LOCATION" \
  --nodepool-name system \
  --node-count "$NODE_COUNT" --node-vm-size "$VM_SIZE" \
  --vnet-subnet-id "$SUBNET_ID" \
  --network-plugin azure --network-plugin-mode overlay --network-dataplane cilium \
  --service-cidr 10.0.0.0/16 --dns-service-ip 10.0.0.10 --pod-cidr 10.244.0.0/16 \
  --enable-managed-identity \
  --enable-keda \
  --enable-addons monitoring --workspace-resource-id "$LAW_ID" \
  --enable-msi-auth-for-monitoring \
  --attach-acr "$ACR" \
  --tags $TAGS \
  --generate-ssh-keys
```
- `--attach-acr`는 kubelet ID에 `AcrPull`을 자동 부여합니다(Terraform의 `aks_acr` 역할 할당과 동일).
- **NAP(Karpenter)** 는 생성 시점에 켜지 않고 **아래 (5)에서 메트릭과 같은 `az aks update` 한 번으로** 켭니다. NAP를 먼저 켜 두면 (5)의 메트릭 `az aks update`가 클러스터를 전체 PUT으로 왕복하며 `nodeProvisioningProfile`을 누락해 **NAP가 기본값 `Manual`로 되돌아갈 수 있기** 때문입니다(단일 PUT로 합쳐 덮어쓰기 방지).
- `--enable-addons monitoring`은 **Container Insights(로그)** 를 켭니다. **Managed Prometheus(메트릭)** 와 **NAP**는 아래 (5)에서 함께 활성화합니다.
- `--enable-msi-auth-for-monitoring`은 워크스페이스 키 대신 **관리 ID(AAD) 인증**으로 온보딩해 로그를 **`ContainerLogV2`** 스키마로 보냅니다(Terraform의 `msi_auth_for_monitoring_enabled = true`와 동일). 이 플래그가 없으면 로그가 레거시 `ContainerLog`(V1)로 가서 모듈 09의 `ContainerLogV2` 쿼리가 빈 결과를 냅니다.

생성 직후, **베스트 프랙티스**로 시스템 노드풀을 시스템 Pod 전용으로 만듭니다(Terraform 경로의 `only_critical_addons_enabled = true`와 동일). taint를 붙이면 앱은 이 풀에 뜨지 못하고 **NAP가 만든 user 노드(`aks-default-*`)** 로 갑니다.
🟢 **실행**
```bash
# 시스템 노드풀 이름 확인(위 az aks create에서 --nodepool-name system 으로 지정)
SYS_POOL=$(az aks nodepool list -g "$RG" --cluster-name "$AKS" --query "[?mode=='System'].name | [0]" -o tsv)
# CriticalAddonsOnly taint 적용 → 시스템 전용
az aks nodepool update -g "$RG" --cluster-name "$AKS" -n "$SYS_POOL" \
  --node-taints "CriticalAddonsOnly=true:NoSchedule"
```
> NAP는 아래 **(5)에서** 켜집니다. 따라서 이 taint를 붙인 직후~(5) 전까지는 앱을 띄울 수 없고(스케줄 불가), (5) 이후 모듈 04에서 앱을 배포하면 NAP가 user 노드(`aks-default-*`)를 만들어 그쪽에 배치합니다.

## (4) 나머지 역할 할당(BYO 서브넷 + Grafana)

`network.tf`/`monitoring.tf`의 역할 할당에 대응.
🟢 **실행**
```bash
# AKS 시스템 ID -> BYO 서브넷 Network Contributor (LB/IP 구성용)
AKS_IDENTITY=$(az aks show -g "$RG" -n "$AKS" --query identity.principalId -o tsv)
az role assignment create --assignee-object-id "$AKS_IDENTITY" --assignee-principal-type ServicePrincipal \
  --role "Network Contributor" --scope "$SUBNET_ID"

# Grafana -> AMW 메트릭 읽기 Monitoring Data Reader
GRAFANA_IDENTITY=$(az grafana show -g "$RG_MON" -n "$GRAFANA" --query identity.principalId -o tsv)
az role assignment create --assignee-object-id "$GRAFANA_IDENTITY" --assignee-principal-type ServicePrincipal \
  --role "Monitoring Data Reader" --scope "$AMW_ID"
# Grafana에 AMW(Managed Prometheus) 통합 연결 — Prometheus 데이터소스가 자동으로 추가됩니다.
# (--skip-role-assignments true: 바로 위에서 Monitoring Data Reader를 이미 부여했으므로 중복 방지.
#  이 명령은 amg 확장 ≥ 2.75.0이 필요하며 최초 실행 시 자동 설치됩니다.)
az grafana integration monitor add -g "$RG_MON" -n "$GRAFANA" \
  --monitor-rg-name "$RG_MON" --monitor-name "$AMW" --skip-role-assignments true

# 실습 사용자 -> Grafana Admin
MY_ID=$(az ad signed-in-user show --query id -o tsv)
az role assignment create --assignee-object-id "$MY_ID" --assignee-principal-type User \
  --role "Grafana Admin" --scope "$GRAFANA_ID"
```

## (5) NAP + Managed Prometheus 메트릭 수집 활성화 (배포 마지막)

인프라가 모두 만들어진 **마지막 단계**에서 **NAP와 메트릭 수집을 한 번의 `az aks update`** 로 함께 켭니다. `--enable-azure-monitor-metrics`는 메트릭 에이전트(`ama-metrics`)와 **수집 파이프라인(DCE/DCR/DCRA + 레코딩 룰)** 을 자동 생성해 `--azure-monitor-workspace-resource-id`로 지정한 AMW로 보내고, `--node-provisioning-mode Auto`는 **NAP(Karpenter)** 를 켭니다(Terraform 경로의 "3) ..." 단계와 동일).
🟢 **실행**
```bash
az aks update -g "$RG" -n "$AKS" \
  --enable-azure-monitor-metrics \
  --azure-monitor-workspace-resource-id "$AMW_ID" \
  --node-provisioning-mode Auto
```
- 클러스터 in-place 업데이트라 수 분 소요됩니다.
- 두 설정을 **같은 update(단일 PUT)** 에 넣어, NAP가 메트릭 업데이트에 덮여 `Manual`로 되돌아가는 것을 방지합니다.
- NAP는 GA라 `--node-provisioning-mode`는 core Azure CLI로 동작합니다. `aks-preview` 확장이 미지원 api-version(`InvalidApiVersionParameter`)을 강제하면 `az extension remove --name aks-preview` 후 재실행하세요.

## (6) 확인

Terraform 경로의 검증과 동일합니다.
🟢 **실행**
```bash
az aks get-credentials -g "$RG" -n "$AKS" --overwrite-existing
kubectl get nodes
kubectl get pods -n kube-system | grep ama-metrics   # ama-metrics-* 가 Running 예상
az aks show -g "$RG" -n "$AKS" --query nodeProvisioningProfile.mode -o tsv   # Auto 예상
# Container Insights '로그' DCR 연결(DCRA) 확인 — MSI 인증에서 ContainerLogV2 수집의 전제.
# 메트릭(MSProm-...)과 로그(MSCI-...) DCRA가 모두 보여야 모듈 09의 ContainerLogV2 쿼리가 채워집니다.
az monitor data-collection rule association list \
  --resource "$(az aks show -g "$RG" -n "$AKS" --query id -o tsv)" -o table
echo "Grafana: $(az grafana show -g "$RG_MON" -n "$GRAFANA" --query properties.endpoint -o tsv)"
```
예상: 시스템 노드 2개가 `Ready`(앱은 `CriticalAddonsOnly` taint로 NAP user 노드에 배치), NAP 모드가 `Auto`, `ama-metrics-*` Pod가 `Running`, DCRA 목록에 메트릭(`MSProm-...`)·로그(`MSCI-...`)가 **모두** 표시.

```text
$ kubectl get nodes
NAME                              STATUS   ROLES    AGE   VERSION
aks-system-12345678-vmss000000    Ready    <none>   3m    v1.34.x
aks-system-12345678-vmss000001    Ready    <none>   3m    v1.34.x

$ az aks show ... --query nodeProvisioningProfile.mode -o tsv
Auto
```
> `nodeProvisioningProfile.mode`가 비어 있거나 `Manual`이면 (5)의 `az aks update`에 `--node-provisioning-mode Auto`가 빠졌거나 별도 update에 덮인 것입니다. (5)처럼 메트릭과 **한 번의 update**에 함께 넣어 재실행하세요.
>
> 💡 **로그 DCRA(`MSCI-...`)가 안 보이면:** az CLI 경로에서는 (2)의 `--enable-addons monitoring --enable-msi-auth-for-monitoring`가 이 로그 DCR/DCRA를 자동 생성합니다(Terraform 경로는 `monitoring.tf`가 명시적으로 생성). 만약 목록에 `MSProm-...`(메트릭)만 있고 `MSCI-...`(로그)가 없으면 MSI 인증에서 로그가 LAW로 흐를 경로가 없어 **모듈 09의 `ContainerLogV2`가 빈 결과**가 됩니다. 복구: `az aks enable-addons -g "$RG" -n "$AKS" -a monitoring --workspace-resource-id "$LAW_ID" --enable-msi-auth-for-monitoring` 실행 후 수 분 뒤 새 로그부터 적재됩니다.

## 이후 모듈 진행 시 (Terraform과의 차이)

모듈 05~07에서 `RG=$(terraform output -raw resource_group_name)`, `AKS=$(terraform output -raw aks_cluster_name)` 로 값을 읽는 부분은, 이 경로에서는 위 (0) 블록에서 설정한 `$RG`·`$AKS` 변수를 **그대로 사용**하면 됩니다(같은 셸 세션 유지). 새 터미널을 열었다면 (0) 블록의 변수만 다시 설정하세요.

**정리(teardown):** Terraform 상태가 없으므로 `terraform destroy` 대신 RG 3종을 직접 삭제합니다. 구체적인 명령과 주의사항은 [10. 정리](10-cleanup.md)의 **[1-A) az CLI로 인프라를 만든 경우](10-cleanup.md#1-a-옵션-az-cli로-인프라를-만든-경우--리소스-그룹-직접-삭제-요청-즉시--백그라운드-수-분십수-분)** 에서 일괄 안내합니다.
